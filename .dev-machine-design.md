## Health Check Configuration

Rust's toolchain catches most issues that required special health checks in other stacks:

- `cargo check` IS the type checker (no Vitest-doesn't-type-check problem)
- `cargo test` includes compilation — test failures ARE build failures
- `cargo clippy` catches lint issues including unused imports, dead code

The health check machine should run:
```
cargo check && cargo test && cargo clippy --all-targets -- -D warnings && cargo fmt --check
```

If any fail, dispatch fix session. Max 10 iterations.

## Working Session Protocol Customizations

### Agent Instruction Additions

Every working session agent for this project:

1. **Read the Python source file** for the module being implemented. Location: `/Users/michaeljabbour/dev/amplifier-foundation/amplifier_foundation/<module>/`
2. **Read the Python test file** for behavioral specification. Location: `/Users/michaeljabbour/dev/amplifier-foundation/tests/test_<module>.py`
3. **Respect the async/sync boundary.** Check the architecture spec section 5 before making any function async.
4. **Use `serde_yaml_ng::Value` for dynamic YAML data.** Never use `serde_yaml` (archived) or `serde_yml` (UNSOUND).
5. **Un-ignore tests as you implement.** Remove `#[ignore = "Wave N"]` when the function is implemented. Tests must pass.
6. **Run `cargo clippy --all-targets` after every feature.** Not just `cargo check`.

### Antagonistic Review Focus Areas

For this Rust port project, reviewers should specifically check:
1. **Async/sync correctness:** Is this function async when it should be sync (or vice versa)?
2. **Error variant matching:** Does the code match on specific `BundleError` variants where Python matches on specific exception types?
3. **Value type correctness:** Is the code using `serde_yaml_ng::Value` vs `serde_json::Value` correctly?
4. **Python behavior fidelity:** Does the Rust implementation match the Python behavior exactly?
5. **Ownership patterns:** Is the code cloning unnecessarily? Are borrows used where possible?
6. **indexmap vs HashMap:** Is insertion order preserved where Python dicts (ordered) require it?
